NLP评价度量指标

NLP评价度量指标

一、引言

​ 在 VTT (Video to Text) 任务中，需要衡量模型输出的视频描述语句的准确性，因为衡量的对象是一个个的自然语言文本，所以通常会选择自然语言处理领域的相关评价指标。这些指标原先都是用来度量机器翻译结果质量的，并且被证明可以很好的反映待评测语句的准确性，主要包含5种：BLEU，METEOR，ROUGE，CIDEr，SPICE.

二、基于词重叠率的方法

2.1、BLEU

​ BLEU(Bilingual Evaluation understudy，双语互译质量评估)是一种流行的机器翻译评价指标，用于分析候选译文中有多少n元词组出现在参考译文中。

感性认识：

​ BLEU算法主要用于判断两个句子的相似程度。拿一句机器翻译的话与其相对应的几个参考翻译作比较，算出一个综合分数。这个分数越高说明机器翻译得越好。

计算公式：

BLEU=BP×exp(∑n=1NWn×logPn)BLEU=BP\times exp(\sum_{n=1}^N W_n \times log P_n) BLEU=BP×exp(n=1∑N​Wn​×logPn​)

BP={1                           lc>lrexp(1−lr/lc)     lc≤lrBP= \begin{cases} 1 \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ \ lc \gt lr\\ exp(1-lr/lc) \ \ \ \ \ lc \le lr \end{cases} BP={1                           lc>lrexp(1−lr/lc)     lc≤lr​

​ lc=机器译文的长度，lr=最短的参考翻译句子的长度

举例：

reference 1： The cat is on the mat.

reference 2： There is a cat on the mat.

candidate 1: the cat the cat on the mat.

bleu的得分有一元组，二元组，三元组等等，这里做了1-3元组的例子，如下：

计算BELU一元组得分:

优点：

  • 它的易于计算且速度快，特别是与人工翻译模型的输出对比；
  • 它应用范围广泛，这可以让你很轻松将模型与相同任务的基准作对比。

缺点：

  • 它不考虑语义，句子结构；
  • 不能很好地处理形态丰富的语句；
  • BLEU 指标偏向于较短的翻译结果（brevity penalty 没有想象中那么强）。

2.2、ROUGE

​ Rouge(recall-oriented understanding for gisting evaluation)是评估自动文摘以及机器翻译的一组指标。

感性认识：

​ 在机器翻译中，将机器生成的候选翻译与人工标注的若干条参考翻译相对比，通过统计二者之间重叠的基本单元（n元语法、词序列和词对）的数目，来评价翻译的质量。

ROUGE 解释
ROUGE-N 基于N-gram 的共现（共同出现）统计
ROUGE-L 基于最长共有子句共现性精度和召回率 Fmeasure 统计
ROUGE-W 带权重的最长共有子句共现性精度和召回率 Fmeasure 统计
ROUGE-S 不连续二元组共现性精度和召回率 Fmeasure 统计

ROUGE-N：

ROUGE−N=∑s∈{ReferenceSummaries}∑gramn∈SCountmatch(gramn)∑s∈{ReferenceSummaries}∑gramn∈SCount(gramn) ROUGE-N=\frac{\sum_{s\in\{ReferenceSummaries\}}\sum_{gram_n\in S}Count_{match}(gram_n)}{\sum_{s\in\{ReferenceSummaries\}}\sum_{gram_n\in S}Count(gram_n)} ROUGE−N=∑s∈{ReferenceSummaries}​∑gramn​∈S​Count(gramn​)∑s∈{ReferenceSummaries}​∑gramn​∈S​Countmatch​(gramn​)​

​ 其中， nnn 表示n-gram， Count(gramn)Count(gram_n)Count(gramn​)表示一个n-gram的出现次数， Countmatch(gramn)Count_{match}(gram_n)Countmatch​(gramn​)表示一个n-gram的共现次数。

例子：

R: there is a cat on the table

C: a cat is on the table

上面例子的 ROUGE-1 和 ROUGE-2 分数如下：

ROUGE−1=∣a,cat,is,on,the,table∣∣there,is,a,cat,on,the,table∣=67ROUGE-1=\frac{|a,cat,is,on,the,table|}{|there,is,a,cat,on,the,table|}=\frac 67 ROUGE−1=∣there,is,a,cat,on,the,table∣∣a,cat,is,on,the,table∣​=76​

ROUGE−2=∣(a,cat),(on,the),(the,table)∣∣(there,is),(is,a),(a,cat),(cat,on),(on,the),(the,table)∣=12ROUGE-2=\frac{|(a,cat),(on,the),(the,table)|}{|(there,is),(is,a),(a,cat),(cat,on),(on,the),(the,table)|}=\frac 12 ROUGE−2=∣(there,is),(is,a),(a,cat),(cat,on),(on,the),(the,table)∣∣(a,cat),(on,the),(the,table)∣​=21​

优点：

  • 直观，简介，能反映词序。

缺点：

  • 区分度不高，且当N>3时，ROUGE-N值通常很小。

ROUGE-L：

ROUGE−L=(1+β2)RlcsPlcsRlcs+β2PlcsROUGE-L=\frac{(1+\beta^2)R_{lcs}P_{lcs}}{R_{lcs}+\beta^2P_{lcs}} ROUGE−L=Rlcs​+β2Plcs​(1+β2)Rlcs​Plcs​​

Rlcs=LCS(X,Y)mR_{lcs}=\frac{LCS(X,Y)}{m} Rlcs​=mLCS(X,Y)​

Plcs=LCS(X,Y)nP_{lcs}=\frac{LCS(X,Y)}{n} Plcs​=nLCS(X,Y)​

​ 其中， XXX 表示候选摘要， YYY表示参考摘要， LCS(X,Y)LCS(X,Y)LCS(X,Y)表示候选摘要与参考摘要的最长公共子序列的长度， mmm表示参考摘要的长度， nnn表示候选摘要的长度。

例子：

R1: Police killed the gunman.

C1: Police ended the gunman.

C2: The gunman murdered police.

C3: The gunman police killed.

ROUGE-L(C1)=3/4

ROUGE-L(C2)=2/4

ROUGE-L(C3)=2/4

通过计算 ROUGE-2(C2)=1/4, ROUGE-2(C3)=2/4, 发现C3优于C2.

优点：

  • 不要求词的连续匹配，只要求按词的出现顺序匹配即可，能够像n-gram一样反映句子级的词序。自动匹配最长公共子序列，不需要预先定义n-gram的长度。

缺点：

  • 只计算一个最长子序列，最终的值忽略了其他备选的最长子序列及较短子序列的影响。

Rouge-W：

​ ROUGE-W 是 ROUGE-L 的改进版，考虑下面的例子

​ R [A B C D E F G]

​ C1 [A B C D H I K]

​ C2 [A H B K C I D]

在这个例子中，明显 C1的翻译质量更高，因为 C1 有更多连续匹配的翻译。但是采用 ROUGE-L 计算得到的分数确实一样的，即 ROUGE-L(R, C1)=ROUGE-L(R, C2)。 因此作者提出了一种加权最长公共子序列方法 (WLCS)，给连续翻译正确的更高的分数，具体做法可以阅读原论文《ROUGE: A Package for Automatic Evaluation of Summaries》

优点：

  • 同一LCS下，对连续匹配词数多的句子赋予更高权重，比LCS区分度更高。

缺点：

  • 同ROUGE-L，只计算一个最长子序列，最终的值忽略了其他备选的最长子序列及较短子序列的影响。

Rouge-S：

​ ROUGE-S 也是对 N-gram 进行统计，但是其采用的 N-gram 允许"跳词 (Skip)"，即跳跃二元组（skip bigram）。例如句子 “I have a cat” 的 Skip 2-gram 包括 (I, have)，(I, a)，(I, cat)，(have, a)，(have, cat)，(a, cat)。最后的分数是通过计算跳词（Skip）匹配数（候选翻译与参考翻译）在参考翻译的跳词数所占的比例获得。

优点：

  • 考虑了所有按词序排列的词对，比n-gram模型更深入反映句子级词序。

缺点：

  • 若不设定最大跳跃词数会出现很多无意义词对。若设定最大跳跃词数，需要指定最大跳跃词数的值。

2.3、METEOR

​ METEOR（Metric for Evaluation of Translation with Explicit Ordering）是用来评估机器翻译效果的度量指标。相比于上面两个标准而言，METEOR 显得更加人性化，它关注到那些翻译准确、但是和候选译文还是对不上的参考译文。在计算方式上它融合了准确率（precision）、召回率（recall），利用二者的调和平均值来作为评判标准。

感性认识：

​ 有时翻译模型是正确的，但它有时和参考翻译不一样（比如适用了同义词），所以我们可以使用WordNet等知识源来扩展同义词集。此外，METEOR还考虑了单词的形式（有着相同词干的单词背认为是部分匹配的，也应该给予奖励，例如将likes翻译成like比翻译成其它的词要好）。

计算公式：

Penalty=γ(chunksunigrams_matched)θPenalty=\gamma( \frac {chunks}{unigrams\_matched})^\theta Penalty=γ(unigrams_matchedchunks​)θ

  • 上式中的 chunks表示匹配上的语块个数，如果模型生成的译文很碎的话，语块个数会非常多；unigrams_matchedunigrams\_matchedunigrams_matched表示匹配上的unigram个数。

Fmean=PmRmαPm+(1−α)RmF_{mean}=\frac{P_mR_m}{\alpha P_m + (1-\alpha)R_m} Fmean​=αPm​+(1−α)Rm​Pm​Rm​​

METEOR=(1−Penalty)FmeanMETEOR=(1-Penalty)F_{mean} METEOR=(1−Penalty)Fmean​

  • Unigram precision P：p=mwtp=\frac m{w_t}p=wt​m​，其中m是候选翻译中，存在于参考翻译中的unigram数。wtw_twt​是候选翻译中的unigram数.

  • Unigram recall R：R=mwrR=\frac m{w_r}R=wr​m​，其中wrw_rwr​是参考翻译中的unigram数.

    通常，取

    α=0.9,γ=0.5,θ=3\alpha =0.9,\gamma=0.5,\theta=3 α=0.9,γ=0.5,θ=3

举例：

Fmean: 1.0000 = 10 × Precision: 1.0000 × Recall: 1.0000 / （Recall: 1.0000 + 9 × Precision: 1.0000）
Penalty: 0.0023 = 0.5 × (Fragmentation: 0.1667 ^3)
Fragmentation: 0.1667 = Chunks: 1.0000 / Matches: 6.0000

优点:

  • 该方法基于一元组的精度和召回的调和平均，召回的权重比精度要高一点 ， 与人类判断相关性高。
  • 引入了外部知识，评价更加友好了。

缺点:

  • 实现非常复杂，目前只有java版本。
  • α、γ和θ 均为用于评价的默认参数。这些都是对着某个数据集调出来的。参数一多听起来就不靠谱。
  • 需要有外部知识。如果很多词不在wordnet，那其实就没什么意义了。

三、Image caption 常用指标

3.1、CIDEr

​ CIDEr（Consensus-based Image Description Evaluation）评价标准是Vedantm在2015年计算机视觉与模式识别大会上提出来的针对图像摘要问题的度量标准。

感性认识：

​ CIDEr 是专门设计出来用于图像标注问题的。这个指标将每个句子都看作“文档”，将其表示成 Term Frequency Inverse Document Frequency（tf-idf）向量的形式，通过对每个n元组进行(TF-IDF) 权重计算，计算参考 caption 与模型生成的 caption 的余弦相似度，来衡量图像标注的一致性的。

  • 是一种加权的评价指标，他更关注你是否说到了重点，而常见的词权重则没有那么高。在 Kaustav_slides image caption的综述里，也提到这个评价指标和人类的评价相关性更高一些。

3.2、SPICE

​ SPICE 也是专门设计出来用于 image caption 问题的，全称是 Semantic Propositional Image Caption Evaluation。

感性认识：

​ SPICE 使用基于图的语义表示来编码 caption 中的 objects, attributes 和 relationships。它先将待评价 caption 和参考 captions 用 Probabilistic Context-Free Grammar (PCFG) dependency parser parse 成 syntactic dependencies trees，然后用基于规则的方法把 dependency tree 映射成 scene graphs。最后计算待评价的 caption 中 objects, attributes 和 relationships 的 F-score 值。

计算公式：

SPICE(c,S)=F1(c,S)=2P(c,S)R(c,S)P(c,S)+R(c,S)SPICE(c,S)=F_1(c,S)=\frac{2P(c,S)R(c,S)}{P(c,S)+R(c,S)} SPICE(c,S)=F1​(c,S)=P(c,S)+R(c,S)2P(c,S)R(c,S)​

P(c,S)=∣T(G(c))⨂T(G(S))∣∣T(G(c))∣P(c,S)=\frac{|T(G(c))\bigotimes T(G(S))|}{|T(G(c))|} P(c,S)=∣T(G(c))∣∣T(G(c))⨂T(G(S))∣​

R(c,S)=∣T(G(c))⨂T(G(S))∣∣T(G(S))∣R(c,S)=\frac{|T(G(c))\bigotimes T(G(S))|}{|T(G(S))|} R(c,S)=∣T(G(S))∣∣T(G(c))⨂T(G(S))∣​

​ 其中，ccc表示候选标题， SSS表示参考标题集合， G(−)G(-)G(−)表示利用某种方法将一段文本转换成一个场景图（Scene Graph）， T(−)T(-)T(−)表示将一个场景图转换成一系列元组（tuple）的集合， 其中的⨂\bigotimes⨂运算类似于交集，与交集不同的地方在于它不是严格匹配，而是类似于METEOR中的匹配。
